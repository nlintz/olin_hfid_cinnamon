Design Overview

Having come up with an initial design of the interface during the paper prototyping stage, our goal in this stage was to create a works-like prototype and use it to further refine our design.

The interface is working to address five major needs that are common to almost all of our users:

  • Simple and intuitive interface
  • Clear visualization of progress
  • Easy access to desired information
  • Continual feedback throughout the process
  • Multiple routes to access information

In this interface, students still have the option to approach course registration from two different views - based on requirements and based on a course list. These two views correspond to the two major screens in the application, the Requirements screen and the Find Classes screen.


The “Requirements” view fulfills the need to visualize progress by providing a list of major and general requirements personalized for the student. The student can easily see what courses have been taken and what courses are left, and can directly add these courses from the requirements list. The student gets instant feedback after clicking the “add” button when the course appears in their “Upcoming Semester” sidebar and appears in the calendar frame on the right. This set of visualization tools communicates the information in multiple ways so that the student can quickly view the proposed schedule as well as visualize major requirements and time conflicts. By combining these three categories in registration, we give students complete autonomy over their schedule as well as verification that they are on track.

The “Find Classes” view gives students easy access to the information they need about the offered courses. Students can search for courses, professors, or acronyms using the search function and get basic information about each course that matches the search query. The student can then click on a button to view more in-depth information about the offered course. The student is able to add courses directly from the “Find Classes” page or from the “More Information” page to give multiple routes to achieve the same result. This means that the system behaves in a way that the user expects, since the repetitive add button means that the student can add a course from whatever view makes the most personal sense. The “Upcoming Semester” sidebar and the calendar frame of the “Requirements” view are again visible in this “Find Classes” view to provide user feedback and promote consistency across the site (adding to the intuitive nature of the site).

Design Walkthrough

The below interaction flow shows a few basic actions using the implemented prototype. Our wireframes and sitemap also present useful visualizations of the layout of and logic behind our site.

Login

When a student first logs into the system, they see the “Requirements” page. The user information is loaded. The basic user profile (name, major, and year of graduation) is shown in the top-left corner, while the user’s current progress on graduation requirements is shown on the general requirements and major requirements panes.

Add Courses

The user clicks the “Requirements” tab in the navigation bar to view a list of required classes, and the view changes to the “Requirements” page. The student can add courses directly from the requirements pane, by clicking the “Add Class” button.

Once the “Add Class” button is clicked, the selected course appears both in the course priority list on the left side and in the calendar view on the right. The “Add Class” button is now grayed out. Clicking the “Drop Class” button removes the class. Dropping the class is also possible by clicking the “X” button on the Course Priority list.

Add Class From Courses

When the student searches for a class by clicking on the search bar at the top of the screen and typing in a query, the student is brought to the “Find Classes” page showing search results.


The student clicks the “Add Class” button to register for the appropriate class and, if applicable, the appropriate section.


The class appears in both the calendar and the “Preferred Courses” sidebar. The student may drop the class by clicking the “Drop Class” button from this screen.

Add Class From The Left Sidebar

The user selects the add class box in the “Course Priority” sidebar on the left and enters a search query.


The first section of the class is added to the “Preferred Courses” sidebar at the bottom and is automatically added to the calendar. “Add class” buttons for courses added this way will be greyed out to reflect the added status.

Add Class From The Class Information Modal

Students can view more information about a particular course by clicking the “learn more” button for each course.


It is also possible to add and drop classes from the “More Information” pop-up screen. After adding the course, the user clicks the “X” button to exit and is returned to the previous page in the same state as it was left in.


Added courses will be shown in the priority sidebar as well as the calendar.



Resolving Time Conflicts

The user notices a time conflict on the calendar and selects the other section, outlined in the same color


The user is switched to the other section and the time conflict is resolved


Changing Course priority

The user clicks the arrows next to courses in the “Preferred Courses” sidebar to order courses according to priority, and the course priority is changed. The priority information is used to help users pre-select alternatives to be automatically enrolled in if top-priority courses are filled up.


The user is switched to the other section and the time conflict is resolved

Major Changes

Our implemented prototype is a result of improvements and modifications on the paper prototype since the Design Development stage. The implemented prototype (a website) accounts for screen size and software limitations and responds to further user considerations which were not apparent in a paper prototype.

The general requirement and major requirement panes on the requirements page were originally conceptualized as two long columns side by side. In the design prototype, however, the requirements are now stacked on top of one another. This was a decision made to deal with the horizontal size constraints of the browser.

In this prototype, where the user does not have all requirements listed, the stacked general and major requirements are fine. In the final prototype we may need to revert back to the original design.

Therefore, there exists a trade-off between the two options: the two-long-columns design allows more information to be reported to the users without forcing them to scroll down much, while the stacked design responds to the limitations of the screen itself.


In the implemented prototype, the user clicks on the “learn more” button instead of just clicking on the course box. The visual cue of the “learn more” button prompts the user that there is more information to be seen and guides the user to the correct action without other instruction. This was a problem that we defined in our cognitive walkthrough of the prototype.


The information displayed in the “More Information” popup is much more condensed due to physical constraints. We chose to get rid of the picture of the professor since it was providing information that students did not require, while taking up much space. The amount of information that is displayed in this popup is still under discussion.

We also added two other ways to close out of this popup besides just the “X” button in the upper right-hand corner. The user can either press the “Close” button in the lower right-hand corner or click outside of the popup. These multiple exit routes prevent the users from getting stuck on the “More Information” popup. This was to address a concern that became apparent while we were paper prototyping, since users would enter the “More Information” popup and not be able to find the way out.


As a response to the Heuristic Evaluation feedback, we have included the “Log Out” button, which previously did not exist in our prototype. This was a crucial flaw in the design where privacy and a sense of closure is key.

Limitations and Future Work

Since work has already started on the second revision, concerns that have already been addressed have been marked as such.

Limitations

UserData

  • The user profile only shows the template profile (major and year of graduation).
  • User cannot see what courses they have already taken because we don’t have that data.
  • You cannot actually register using the site.

Database

  • The list of courses is incomplete.
  • Data on website is based on Olin class data but somewhat manufactured.
  • Prerequisite data is not included.
  • Data specific to sections is not always available (for instance, if sections are taught by different professors).
  • The user cannot sign up for different sections of the same class (which becomes an issue when ordering courses according to priority).
  • This application is not connected to the Olin Registrar in any way.

Searching

  • Search query does not autocomplete.
  • “Add Class” function under course priority sidebar can give unexpected results.

Styling

  • Formatting and GUI are not styled perfectly.
  • Site navigation bar is not affixed to the top of the screen.
  • Site is not completely responsive to changing window sizes.
  • Courses are not alphabetized in full course list.

Insights

We have learned a few key things throughout this phase of the project. We abstracted these to a heuristic followed by an explanation of the heuristic.


Keep in mind time constraints. Software functionality takes time to implement, so it is important to decide early what should be fully implemented and what can be downsized. We had a number of features that were either not fully operational or not implemented because of time constraints. For example, we wanted our search to behave a lot like Google instant, but we didn’t have the time to implement that interaction.

Plan around physical constraints. The size of a normal screen puts a limit on how much information an interface can contain. For example, we wanted to have multiple requirement columns for better visualization but needed to cut that functionality due to screen size limitations.

Test early and often. Some of our best insights came from having peers or potential users test our system, especially about things that we didn’t design for. We have included a number of these in our modifications for Revision 2 and included the rest in the limitations section.

Consider different affordances between paper and electronic mediums. When we paper prototyped, many users noticed things immediately that were less obvious on an electronic medium. For example, they knew to click the bordered sections to change sections in the calendar on the paper prototype but missed it in the electronic medium. This is because web sites afford clicking through explicit labels or buttons whereas paper prototypes have less explicit affordances opening up users to explore more interactions. We believe that by following the design patterns from the web in paper prototyping, user feedback will be more natural and reflect how people will actually interact with your site.

Focus on the main design and worry about edge cases after. We should spend the majority of the time during implementation making the parts of the site that all users will see on a regular basis. While edge cases are important to consider, they will not make or break a design, so first priority should be given to the design of the main site. Even while keeping this in mind, it was difficult for us to not get wrapped up in edge cases, especially when we first came across them.

Aesthetics are not completely separable from usability. Initially we had only focused on implementing features such as the business logic which powers the calendar widget. However, we de-prioritized styling and aesthetics when developing this widget which made it look strange and out of place on the site. By adding some simple colors and rounded edges, we made the calendar fit within the general design of the site and made it much simpler to understand. Therefore, focusing on aesthetics can actually make a feature more usable especially when the widget is trying to convey a complex idea like resolving conflicting course times.
