Design Overview

Having come up with an initial design of the interface during the paper prototyping stage, our goal in this stage was to create a works-like prototype and use it to further refine our design.

The interface is working to address five major needs that are common to almost all of our users:

  • Simple and intuitive interface
  • Clear visualization of progress
  • Easy access to desired information
  • Continual feedback throughout the process
  • Multiple routes to access information

In this interface, students still have the option to approach course registration from two different views - based on requirements and based on a course list. These two views correspond to the two major screens in the application, the Requirements screen and the Find Classes screen.


The “Requirements” view fulfills the need to visualize progress by providing a list of major and general requirements personalized for the student. The student can easily see what courses have been taken and what courses are left, and can directly add these courses from the requirements list. The student gets instant feedback after clicking the “add” button when the course appears in their “Upcoming Semester” sidebar and appears in the calendar frame on the right. This set of visualization tools communicates the information in multiple ways so that the student can quickly view the proposed schedule as well as visualize major requirements and time conflicts. By combining these three categories in registration, we give students complete autonomy over their schedule as well as verification that they are on track.

The “Find Classes” view gives students easy access to the information they need about the offered courses. Students can search for courses, professors, or acronyms using the search function and get basic information about each course that matches the search query. The student can then click on a button to view more in-depth information about the offered course. The student is able to add courses directly from the “Find Classes” page or from the “More Information” page to give multiple routes to achieve the same result. This means that the system behaves in a way that the user expects, since the repetitive add button means that the student can add a course from whatever view makes the most personal sense. The “Upcoming Semester” sidebar and the calendar frame of the “Requirements” view are again visible in this “Find Classes” view to provide user feedback and promote consistency across the site (adding to the intuitive nature of the site).

Documentation of Limitations

User Data
  • The user profile only shows template profile (major and year of graduation)
  • User cannot see what courses they have already taken because we don’t have that data.
  • You cannot actually register using the site.
Database
  • The list of courses is incomplete
  • Data on website is manufactured
  • Prerequisite data is not included
  • Data specific to sections is not always available (for instance, if sections are taught by different professors).
  • This application is not actually connected to the Registrar in any way.
Search features
  • Search query does not autocomplete.
  • Search queries look through all of the course information.
  • Add courses function under course priority sidebar can give unexpected results.
Styling
  • Formatting and GUI are not styled perfectly.
  • Site navigation bar is not affixed to the top of the screen.
  • Site is not completely responsive to changing window sizes.
  • Courses sometimes show up in the same color on the calendar (since we’re currently only pulling from a list of eight colors).
  • Courses on course priority list are not color-coded to match courses in calendar.
  • Courses are not alphabetized in full course list.
